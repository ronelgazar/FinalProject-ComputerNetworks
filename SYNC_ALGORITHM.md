# אלגוריתם הסנכרון — ניסוח מדויק

כדי להתמודד עם זמני השהיה שונים בין המשתמשים, המערכת מבצעת תחילה איסוף של מדדי RTT מכלל הלקוחות. כל לקוח שולח בקשת `schedule_req` הכוללת את הערכים `rtt_ms` ו־`handshake_rtt_ms` שנמדדו בשלב ה־handshake. בנוסף, לפני שלב ה־`exam_req`, הלקוח שולח עדכון RTT טרי המבוסס על 12 סבבי `time_req/time_resp`. הלקוח מסכם את המדידות לשלושה ערכים מייצגים (`rtt_min`, `rtt_med`, `rtt_max`) ושולח אותם בשדה `rtt_samples`. השרת אוסף את כלל הדגימות במשך חלון זמן של 1.5 שניות.

לאחר סגירת חלון האיסוף, השרת מחשב עבור כל לקוח את זמן ההגעה החד־כיווני המשוער (D_i), ובהתאם לכך את זמן השליחה שלו. זמן השליחה של כל לקוח מחושב ביחס לזמן הגעה משותף מתוכנן (T_arr), לפי:

```
T_send,i = T_arr − D_i
```

כאשר זמן ההגעה המשותף מעוגן לערך הקבוע מראש:

```
T_arr = start_at_ms
```

במקרה שבו `start_at_ms` קרוב מדי לזמן הנוכחי או שכבר חלף, המערכת משתמשת במנגנון fallback דינמי:

```
T_arr = t_now + max_i(D_i) + N·C + M
```

כאשר:

* N — מספר הלקוחות המחוברים
* C — תקורת שליחה משוערת לכל לקוח
* M — מרווח בטיחות תזמוני קטן

הערך `start_at_ms` נשמר בקובץ משותף לכל מופעי השרת (`/app/shared/start_at.txt`), ולכן כל השרתים עובדים מול אותו זמן יעד.

---

## חישוב D_i — פר־לקוח

חישוב D_i מתבסס על שלוש דגימות RTT של הלקוח עצמו, הנגזרות מתוך 12 סבבי הסנכרון. השרת מקבל מהלקוח את הערכים `rtt_min`, `rtt_med`, `rtt_max` ומחשב:

```
s       = sorted([rtt_min, rtt_med, rtt_max])
RTT_med = s[1]
J_rtt   = s[2] − s[0]
```

כאשר:

* RTT_med — החציון של זמני ה־RTT עבור אותו לקוח
* J_rtt — טווח הג'יטר של המדידות

אם נמדד עבור הלקוח ערך היסט שעון offset_i (תיקון א־סימטרי של זמני העלייה והירידה), מחשבים תחילה את זמן הירידה המשוער:

```
d_down,i = RTT_med / 2 − offset_i
D_i      = max(1, d_down,i + J_rtt / 2)
```

אם אין מידע על היסט השעון, מניחים נתיב סימטרי ומחשבים:

```
D_i = (RTT_med + J_rtt) / 2
```

במקרה שאין כלל דגימות זמינות, משתמשים בהערכה בסיסית:

```
D_i = rtt_ms / 2
```

באופן זה, לקוח עם RTT גבוה יותר או jitter גבוה יותר יקבל D_i גדול יותר. לכן החבילה עבורו תישלח מוקדם יותר, כך שזמן ההגעה לדפדפן יתכנס ככל האפשר ל־T_arr עבור כלל הלקוחות.

---

## סדר השליחה

הלקוחות נשלחים לפי סדר עולה של T_send,i (כלומר לפי D_i יורד), מהשליחה המוקדמת ביותר למאוחרת ביותר. ההפרש בין שליחה ללקוח k לבין הלקוח הבא מחושב לפי:

```
Δ_k = max(0, T_send,k+1 − T_send,k − C)
```

כאשר C מייצג את תקורת השליחה המשוערת לכל לקוח.

## Bridge Self-Correcting Hold

בנוסף לכך, בצד ה־bridge מופעל מנגנון החזקה קצר לצורך יישור זמן ההגעה לדפדפן. השרת מחתים כל חבילה בזמן השליחה:

```
server_sent_at_ms = time.time() * 1000
```

כאשר הגשר מקבל את החבילה, הוא מחשב את זמן היעד להעברה לדפדפן:

```
target_ms = server_sent_at_ms + max_rtt / 2 + JITTER_BUFFER_MS
hold_ms   = max(0, target_ms − t_now)
```

כאשר:

* `max_rtt` — ערך ה־RTT המקסימלי שנמדד בין כלל הלקוחות
* `JITTER_BUFFER_MS = 5 ms`

אם שידור חוזר של RUDP עיכב את החבילה ב־Δ מילישניות, אז הזמן הנוכחי t_now כבר התקדם באותו פרק זמן. לכן ערך `hold_ms` מתקצר בהתאם — מנגנון תיקון עצמי המאפשר לשמור על זמן יעד אחיד גם כאשר מתרחשים עיכובים או שידורים חוזרים ברשת.

---

## תוצאה

באופן זה, המערכת מחשבת בפועל את זמן השליחה של כל לקוח כך שזמן ההגעה לדפדפן יהיה מסונכרן והוגן ככל האפשר. כל לקוח מקבל את החבילה לדפדפן בפרק הזמן:

```
target_ms ± 5 ms
```

כלומר, ללא תלות מהותית ב־RTT האישי של הלקוח, ובסטייה של מספר מילישניות בודדות בלבד.
